The CRT Framework

Jay's organizational structure for setting up your creative AI agent workspace. Works across Claude Code, Antigravity, and Codex.

C — R — T

C

.Claude / .Agent

Your AI agent's brain. A markdown file (CLAUDE.md or .agent) containing all instructions, directives, API key references, provider system configuration, and personality. This is the orchestrator that controls everything.

R

References

Your inputs, brand images, style references, product shots, and prompt best practices documentation (387+ lines). Everything the agent needs as creative fuel. Continuously updated as you refine your style.

T

Tools

Python scripts and workflows that route tasks to different AI models. Handles Airtable updates, image/video generation API calls, cost tracking, and provider failover. Think of each .py file as an n8n workflow in code form.

The Provider System inside CLAUDE.md lets you override which API aggregator handles generation by passing provider="kie", provider="google", or provider="wavespeed" to generation functions.

CRT in Practice: Zev's Live Implementation

The actual CRT files live in crt/ at the workspace root. Here is what each contains:

FileContents
C-agent.mdCreative director identity, full Provider Registry with 5 models (Nano Banana 2, Nano Banana Pro, FLUX.2 Pro/Max/Flex), API model strings, prompt format rules per model, cost tracking, default routing table
R-references.mdPrompt engineering best practices per model (natural language for Nano Banana, JSON structured for FLUX), art style guides (Chibi, Pixel Art), sprite sheet specs, color palettes, character consistency checklists, file naming conventions
T-tools.mdPython routing code with PROVIDERS dict and ROUTING table, Airtable CRUD helpers, prompt generators (natural language + JSON), cost tracking per provider, environment variable setup

Key Insight: Natural Language vs JSON Prompts

Nano Banana 2/Pro: Natural language prompts only. Descriptive sentences, plain English colors, character descriptions every time. No hex codes.

FLUX 2 Pro/Max/Flex: JSON structured prompts preferred. Hex color codes supported and recommended for precision. Subject-first hierarchy.

Project extensions: Each project gets a PROJECT-*.md file with project-specific anchor strings, character details, and color palettes. Global CRT files never change per project - only the PROJECT file swaps.

Tips and Lessons Learned

  • LaunchAgents vs. in-session crons: LaunchAgents persist across reboots and run outside Claude sessions via claude --print. In-session crons (CronCreate) expire after 3 days and only work while a session is active. Use both for redundancy.
  • macOS provenance xattrs: Scripts downloaded or copied via Finder get com.apple.provenance extended attributes that block execution from launchd (exit code 126). Fix with xattr -d com.apple.provenance script.sh.
  • claude --print cannot nest: Running claude --print inside an existing Claude Code session fails with "cannot be launched inside another Claude Code session." This is expected. LaunchAgent scripts work correctly when launched by launchd (no parent session).
  • Skills need mirroring: Skills are read from both ~/.claude/skills/ (global) and project-level .claude/skills/. Keep both in sync, or use symlinks via setup.sh.
  • MEMORY.md has a 200-line soft limit: Lines after 200 may be truncated in context. Keep it concise and offload details to topic files linked from MEMORY.md.
  • ~/.claude.json gets modified externally: Claude Code itself updates this file (stats, usage). Always re-read before editing to avoid "file modified since read" errors.
  • Storyboard prompts, model-specific color rules: Nano Banana Pro/2 renders hex codes as literal swatches - use plain English. FLUX 2 Pro supports and benefits from hex codes for precision.
  • HTTPS only for GitHub: SSH keys with passphrases fail in automated contexts. Always use HTTPS remotes (https://github.com/user/repo), not SSH (git@github.com:). The gh CLI authenticates via HTTPS token automatically.
  • OG images + favicons are mandatory: Every website gets Open Graph images (1200x630px per page) and multi-size favicons. Links without preview images look unprofessional. Build these in from the start, not as an afterthought.
  • /compact is useful: Use it proactively at 50-60% context usage. CLAUDE.md and memory files survive compaction (they reload from disk). With the 1M context window on Opus 4.6, you have more runway than before.

10.3 Watchdog Script

A cron-based watchdog that runs outside the agent process. Every 5 minutes, it checks if the gateway is healthy. If it's down, it validates the config (restoring from golden backup if corrupted) and restarts.

#!/bin/bash
# ~/.openclaw/watchdog.sh
LOGFILE="$HOME/.openclaw/logs/watchdog.log"
GOLDEN="$HOME/.openclaw/openclaw.golden.json"
CONFIG="$HOME/.openclaw/openclaw.json"
mkdir -p "$HOME/.openclaw/logs"

if ! curl -s --max-time 5 http://127.0.0.1:18789/health > /dev/null 2>&1; then
  echo "$(date '+%Y-%m-%d %H:%M:%S') Gateway down, attempting restart..." >> "$LOGFILE"
  if python3 -c "import json; json.load(open('$CONFIG')); print('valid')" 2>/dev/null | grep -q valid; then
    openclaw gateway restart >> "$LOGFILE" 2>&1
  else
    echo "$(date '+%Y-%m-%d %H:%M:%S') Config invalid! Restoring golden backup." >> "$LOGFILE"
    cp "$GOLDEN" "$CONFIG"
    openclaw gateway restart >> "$LOGFILE" 2>&1
  fi
fi

Install to crontab:

chmod +x ~/.openclaw/watchdog.sh
(crontab -l 2>/dev/null; echo "*/5 * * * * bash $HOME/.openclaw/watchdog.sh") | crontab -

Appendix C: Troubleshooting

Config corrupted / JSON parse error:

  1. Check what changed: cd ~/.openclaw && git diff
  2. If minor: fix the JSON manually in VS Code (never nano with voice input)
  3. If major: restore golden backup: cp ~/.openclaw/openclaw.golden.json ~/.openclaw/openclaw.json
  4. Restart: openclaw gateway restart
  5. Commit the fix: cd ~/.openclaw && git add openclaw.json && git commit -m "restored from golden"

Model not found error:

  1. Run openclaw models list to see what's in the catalog
  2. If the model shows as "missing," it's not in OpenClaw's catalog yet
  3. Use openclaw models set with a supported model
  4. Run openclaw update and check again
  5. Google model names are exact: gemini-3.1-flash-lite is not gemini-3.1-flash

Rate limit / cooldown:

  1. Run openclaw models status to see cooldown timers
  2. Switch to a different provider temporarily: openclaw models set anthropic/claude-sonnet-4-6
  3. Switch back after cooldown clears
  4. Consider adding multiple providers to your fallback chain to avoid this

Agent crashed after self-update:

  1. The agent tried to update its own binary while running. This kills the process.
  2. Run the update yourself: openclaw update
  3. Reinstall the LaunchAgent: openclaw gateway install
  4. Start: openclaw gateway start
  5. Add rules to SOUL.md preventing self-updates (see Phase 10.2)

Gateway not running after update:

  1. Updates can change the entrypoint file path
  2. Run openclaw doctor and accept the gateway service config update
  3. Then: openclaw gateway install && openclaw gateway start
Appendix D: Lessons Learned
  1. Never let the agent self-update. Running openclaw update replaces the binary the agent is running on. It's like swapping an engine while driving. The agent should provide commands for the operator to run manually.
  2. Never edit JSON with nano + voice input tools. Superwhisper, macOS dictation, and similar tools can paste content multiple times into nano, duplicating the entire file. Use VS Code or openclaw models set CLI commands.
  3. Use CLI commands over manual JSON editing. openclaw models set, openclaw models fallbacks add, and openclaw models aliases are safer than hand-editing openclaw.json.
  4. Google model names are precise. gemini-3.1-flash does not exist. It's gemini-3.1-flash-lite. New models may not be in the OpenClaw catalog for days after Google releases them.
  5. Rotate keys immediately if exposed. During troubleshooting, config files with API keys can end up in chat logs, pastes, or screenshots. Rotate all credentials afterward.
  6. Install stability safeguards on day one. The golden backup, watchdog, and git tracking should be set up immediately after first successful boot, not after the first crash.
  7. Two Git repos, not one. ~/.openclaw/ gets a config-tracking repo. ~/.openclaw/workspace/ gets the brain backup repo (mr-anderson-brain). They serve different purposes and must stay separate.
  8. OpenClaw doctor can overwrite your config. Running openclaw doctor --fix may reset settings. Always have a golden backup before running it.
  9. Preview models have lower rate limits. Even on paid Google tiers, preview models (like gemini-3.1-pro-preview) have tighter limits than stable GA models.
  10. Set exec approvals before giving the agent tasks. Default settings gate every shell command behind a Terminal prompt.

Knowledge & Security

Two protocols that compound over time: an LLM-maintained knowledge base (inspired by Andrej Karpathy) and a secrets management system that eliminates plaintext credentials from your machine. Both were implemented April 2026 and are now part of every session.

Source: Andrej Karpathy's viral tweet on LLM Knowledge Bases (18M+ views, April 2026) + Electra Filmworks security audit

The Karpathy Knowledge Base Pattern

Andrej Karpathy (former head of Tesla AI, co-built GPT at OpenAI) shared how he uses LLMs to build personal knowledge bases for research topics. The core insight: instead of asking an LLM questions and letting the answers evaporate, you compile them into a structured wiki that the LLM maintains. Knowledge compounds across sessions instead of resetting every time.

1

The Compilation Loop

This is the pattern that makes it work:

  1. Raw ingestion - Source material (articles, repos, research, client data) goes into project folders
  2. LLM compilation - The LLM structures raw data into linked markdown articles in a Knowledge directory. The human rarely edits directly.
  3. Cumulative Q&A - When a complex question is researched, the answer gets filed back into the knowledge base. Every query adds to the system.
  4. Linting - Periodic LLM passes find gaps, stale info, inconsistencies, and connections between articles
  5. The wiki is the LLM's domain - You read and benefit. The LLM writes and maintains. You are the reader, not the editor.
